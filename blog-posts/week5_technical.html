class HowToRubyClasses

By now, chances are that you’ve created several methods that generally work independently to - oh I don’t know, separate numbers with commas or create accountability groups. As you begin solving larger challenges, if you find that you are using the same method to run on a number of different objects, you would probably benefit from using a class. Granted you can always accomplish this using if/else, unless, or case statements to achieve the same result, but it will be at the cost of your codes organization and I’m sure you will be thankful you didn’t when the time comes to debug.

In the most generic terms, classes are templates for creating object instances. Okay, so what do I mean by that? Let zoom out from the programming world for a moment and refer to objects as we define them in the ‘real world.’ Everything around us is an object. A desk, a mug, a computer, a DVD (& yes, at this point I’m just naming things in a two foot radius of myself- super creative, I know!) Each object has it’s own attribute,for instance: desk (color, size, material) or DVD or movie ( genre, director, title). Each object is simply an instance of a class and shares all the same attributes of what is it to be a desk or a movie. Once it became clear to me that everything in Ruby is also an object, I realized that we’ve been dealing with classes from the beginning.

Switching back to the programming world, if you can see that Array is an object instance of a class and gives you access to the method #each(among other methods), then you pretty much know how to instantiate new objects.

my_array = Array.new
=>[]

Look familiar?

In order to better explain classes, let's create our very own.

Why don't we create a class for movie? When we create a class Movie, we want to make sure that we also provide a method to initialize which can also hold that objects attributes. This way, when you call the ::new method on the class Movie, it creates an instance of Movie which automatically runs your #initialize method setting up that Movie for use.

class Movie
    def initialize (genre, director, title)
      #code
    end
end

Although some movies will share the same genre and/or director, it is unlikely that your movies will share a title. Turns out classes not only share their methods, but can also share their variables. In order to make sure that each movie will have a variable that functions the same way in your constructed class, we are given access to instance variables This way you’ll be able to use the variable titlethe same way for every instance of Movie while holding its own unique value or title in this case. In order to differentiate between variables and instances of them, we differentiate them using @instance_var_name notation. These are also usually set up in your #initialize method like so:

class Movie
    def initialize (genre, director, title)
      @genre = genre
      @director = director
      @title = title
    end
end

These instance variables help set up your Movie’s state and when that specific movie or instance is deleted, so are the associated instance variables.

Cool, so now you have a way to build a catalog of all your movies. Now that you have an efficient way to store your data, we are also given a way to make our Movie objects operate specified actions. Since these methods get called on an individual instance of the Movie class, they are referred to as instance methods. They work exactly like #each, #map, and #sort would for an instance on the Array class. These methods look like any other method; beginning with a def and ending with end .

class Movie
    def initialize (genre, director, title)
      @genre = genre
      @director = director
      @title = title
    end

    def get_genre(genre)
      puts @genre
    end
end
